Black Box Replay: scrub through your whole build, turn by turn.

One self-contained HTML file — no server, no network — assembled entirely from what Supersonic already writes to disk: every turn's diff, its full Verify gate breakdown, its Signed Turn Receipt, and any rule the Rules Engine learned. Every receipt's signature is verified server-side, with the exact function sonic verify-receipts uses. Every turn's diff hash is independently re-derived in your own browser via the Web Crypto API — but only where the page-generation step has already proven that recomputation is honest: it hashes the diff it reconstructed and checks it against the receipt's own hash before ever showing the button. A turn with a rollback before it shows the diff for reading with a plain note about why the button isn't offered, instead of a badge that's silently wrong.

Secret Leak Gate: hardcoded credentials never ship.

Every turn's added diff lines are scanned for the structural shape of a real credential — AWS keys, PEM blocks, GitHub/Slack/Stripe/Anthropic/OpenAI/Google tokens, a new committed .env file. A high-confidence match fails the turn outright, the same severity as a syntax error. Known placeholder conventions are filtered out first, so an .env.example template doesn't trip it on every turn.

Test Quality Gate: a green check that means something.

Once a turn's real tests pass, a small bounded set of AST-level mutants (comparison/boolean/constant flips) scoped only to the functions it touched are re-tested against the same suite. A mutant the suite doesn't catch is a test that passes without actually verifying that logic — the one failure mode "tests: PASS" cannot see by definition. A soft signal, not a veto: it votes alongside the other seven, it doesn't fail a turn on its own.

PTY-native execution: your coding agent gets a real terminal.

An optional execution mode that runs the coding-agent CLI inside a genuine pseudo-terminal (POSIX pty.fork()) instead of a plain subprocess pipe — so a CLI that special-cases "not attached to a terminal" (disabling colors, progress bars, interactive confirmations) behaves the same way it would if a human had typed the command. Falls back to the standard subprocess path automatically on any platform or CLI it can't run cleanly on. To be precise about what a PTY actually is: it governs a process's stdin/stdout, not its filesystem writes — Supersonic doesn't claim otherwise, and doesn't pretend to intercept a file write before it hits disk. That's a different, much heavier mechanism (ptrace/seccomp, or a FUSE filesystem) that this feature is not.

Docker Sandbox: contain the blast radius, not just the workdir.

Checkpoint/rollback already protects a project's own workdir with a hard git reset — but that's the only thing it protects. A coding agent's shell can reach anywhere the host OS user can: another project's folder, credentials in your home directory, anything. Docker Sandbox runs the coding-agent CLI inside a throwaway container with only that one workdir bind-mounted in, all Linux capabilities dropped, and memory/CPU/PID limits enforced, so that blast radius is actually contained. To be precise about scope: this does not sandbox network egress — the agent still needs outbound access to its own LLM provider's API — and it's not zero-config the way PTY mode is: it requires Docker installed and a real image built ahead of time. Falls back cleanly to PTY or plain-subprocess execution if Docker isn't reachable or no image is configured.

Live Syntax Watch: sees a broken file before the turn ends.

A concurrent background thread — mtime polling plus ast.parse, no kernel hooks — that re-checks every touched Python file within a fraction of a second of it being saved, while the agent is still mid-turn. Syntax Shield's diff-based check still runs as the authoritative gate after the turn completes; this is faster feedback in front of it, not a replacement. Observability only in this version — it surfaces the exact file and line early, it does not pause or interrupt the agent process itself.

Self-Evolving Rules Engine: the same mistake, once.

When the exact same Verify failure category repeats across turns, a supervisor-critic LLM call synthesizes one concise, durable rule from the specific failure trace and appends it to the project's own .supersonic/rules.md — folded into every subsequent turn's prompt. Best-effort, it also mirrors into .cursorrules or CLAUDE.md if the project already has one, updating only its own clearly marked section and leaving everything else untouched — never creating one of those files from scratch. This is a narrower mechanism than genetic-algorithm prompt mutation: one durable rule per repeated failure, not a population of candidate prompts scored against a fitness function.

Multi-Repository State Anchoring: one feature, many repos, one rollback.

Register the other repos a feature ticket actually spans — a frontend, a backend, a schema-definitions repo — and every turn snapshots all of them alongside the primary repo. When the primary repo's turn ships, every linked repo's checkpoint refreshes to match; when it fails Verify and rolls back, every linked repo rolls back with it. The coding agent still runs exactly once per turn, against the primary workdir only — what's coordinated here is checkpoint and rollback, so nothing in the linked set can silently drift out of sync with what the primary repo's Verify gate actually approved.

How it works

1 · Plan

Ground the idea, write the plan

Optional research enrichment, then one call produces a build plan and a name — provider-agnostic, no vendor lock-in.

2 · Checkpoint

Git-native snapshotting

Every verified state is committed and tagged. No custom snapshot format — just git doing what it already does well.

3 · Build

Your agent writes code

Claude Code, Codex, OpenCode, Cursor Agent, or Aider — bring your own CLI. Optionally runs inside a real PTY (pty.fork()) instead of a plain pipe, or sandboxed inside a throwaway Docker container with only the workdir mounted in — falls back to the standard subprocess path automatically wherever either can't run.

4 · Live Watch

Catches a broken file while the agent is still writing

A concurrent thread re-parses each touched .py file within a fraction of a second of it being saved, mid-turn — not after the whole turn ends. Observability only: it surfaces the exact file and line early, it does not interrupt the agent process.

5 · Dependency Trust

Every new package checked before it ships

Newly-added dependencies are verified against the real PyPI/npm registry. A package that doesn't exist fails the turn — one corrective re-prompt, then rollback, before it ever reaches the expensive gate below.

6 · Secret Leak

Every diff scanned for hardcoded credentials

Added lines are checked against real credential formats (AWS, GitHub, Slack, Stripe, Anthropic, OpenAI, Google, PEM keys) and new .env files. A high-confidence match fails the turn — one corrective re-prompt, then rollback, same as Dependency Trust.

7 · Test Quality

Mutation-tested, not just test-passed

Once the real suite passes, a bounded set of AST mutants scoped to the functions this turn touched (comparison/boolean/constant flips) are re-run against it. A mutant the suite doesn't catch means a test that passes without verifying that logic — a fair-vote signal, not a veto.

8 · Verify

Eight independent signals

Tests, lint/typecheck, an LLM goal-satisfaction critic, a diff-similarity thrash detector, an optional browser telemetry check, mutation-based test quality, and the Dependency Trust and Secret Leak results above.

9 · Signed Receipt

An Ed25519-signed reproducibility record

Prompt hash, diff hash, and the full gate verdict, signed and written into the same commit as the checkpoint below. Verify any receipt offline with sonic verify-receipts — no access to the machine that generated it required.

10 · Rollback or ship

Only proven progress sticks

Pass → new checkpoint, pushed to GitHub — and every linked repo (see Multi-Repo Anchoring below) refreshes its own checkpoint to match. Fail → hard reset to the last good state in every one of them together, failure logged to the Continuity Graph.

11 · Review Risk

Told what to actually check

Every shipped turn's changed files ranked by blast radius, sensitive-path exposure, missing test coverage, unverified dependencies, and suspicious credential-shaped assignments — a static heuristic, not another LLM call, so it's instant.

↻ · Escalation

A HIGH-risk turn upgrades the next one

Review Risk flagging a HIGH file feeds back into Build and Verify: the very next turn runs at the stronger model you've configured for that coding agent and for the critic, then falls back to the default the turn after. Cross-turn, not a stage — it never fires with no risky-file history, and it never overrides a model you haven't opted in.

↻ · Rules Engine

The same failure, learned once

When one Verify failure category repeats across turns, a supervisor-critic call writes one durable rule into the project's own rules file — folded into every prompt after that, and mirrored into .cursorrules/CLAUDE.md if the project already has one. One rule per category, appended, never rewritten.

↻ · Multi-Repo Anchoring

Frontend, backend, schema — rolled back together

Register the other repos a feature ticket spans. Every turn snapshots all of them alongside the primary repo; a passed turn refreshes every linked checkpoint, a failed one rolls every linked repo back with it — the agent still runs once per turn, against the primary workdir only.

FAQ

What is Review Risk, actually?

A static heuristic (no LLM call) that runs once a turn ships. It parses the diff, counts how many other files import each changed file (blast radius), checks changed lines and paths against a list of sensitive-surface keywords (auth, payment, permissions, migrations, deletion), and checks whether a matching test file was touched in the same turn. Files score high/medium/low and get a plain-English reason. It has false positives and false negatives — the goal is redirecting a human's five minutes of attention, not replacing judgment.

What is the Dependency Trust Gate?

A pre-flight check that runs before the expensive Verify gate. It reads the turn's diff for any newly-added package (manifest entries in requirements.txt/pyproject.toml/Pipfile/package.json, or a raw install-command line), and checks each one against the real PyPI or npm registry. A package that doesn't exist fails the turn outright — that's "slopsquatting": an attacker pre-registers a name an AI agent is statistically likely to hallucinate, ships malware under it, and waits. It can't stop a malicious install from executing once inside a single turn's working tree (Supersonic doesn't sandbox the coding agent's own shell commands), but it guarantees that dependency never gets checkpointed, shipped, or pushed to GitHub — and the agent gets one corrective re-prompt naming the exact bad package first.

What is the Secret Leak Gate?

A pre-flight check, same family as Syntax Shield and the Dependency Trust Gate, that runs before the expensive Verify gate. It scans only the added lines of a turn's diff for the structural shape of a real credential: AWS access key IDs, PEM private key blocks, GitHub/Slack/Stripe/Anthropic/OpenAI/Google token formats, and Slack webhook URLs. A match on any of those formats is specific enough to be a real secret, not a coincidence, and fails the turn outright — one corrective re-prompt naming the exact file and pattern, then rollback if it's still there. A new or modified real .env file (not .env.example) is flagged the same way regardless of content. A separate, noisier heuristic — a variable name like api_key or password assigned a long opaque string — is surfaced in the Review Brief instead of blocking, since it's prone to false positives on things like test fixture IDs. GitGuardian's State of Secrets Sprawl 2026 report found AI-assisted commits leak a real secret at roughly double the rate of human-only commits (~3.2% vs. ~1.6%), and that 64% of secrets confirmed valid in 2022 were still exploitable in January 2026 because nobody rotated them — this gate exists to stop that pattern before it starts.

What is the Test Quality Gate?

A check that catches the one failure mode "tests: PASS" can't see by definition: a test that runs the code but never actually verifies anything. Once a turn's real test suite passes, the gate parses the diff to find which functions were touched, generates a small bounded set of single-point AST mutants against just those functions (comparison operator flips, boolean operator flips, boolean constant flips — capped at 4 per function, 16 total, inside a 90-second budget), and re-runs the same test command against each mutant in turn, restoring the original file immediately after every run. A mutant the suite still passes against is a "survivor" — evidence that whatever line changed, no test noticed. Unlike Dependency Trust and Secret Leak, this is a fair-vote signal, not a veto: a surviving mutant is heuristic evidence of a weak test, not proof of a bug, so it participates in the normal Verify signal count instead of failing the turn outright.

What are Signed Turn Receipts?

An Ed25519-signed JSON attestation, written for every turn that ships, to .supersonic/receipts/turn-<n>.json — in the same git commit as the checkpoint it describes. It records a SHA-256 hash of the literal prompt sent to the coding agent (including whichever corrective re-prompt, if any, actually produced the shipped diff), a SHA-256 hash of the diff itself, a files/insertions/deletions stat, which LLM provider/model Supersonic's own loop used that turn, which coding-agent CLI wrote the diff, and the full Verify gate verdict — every signal, not just pass/fail. The private signing key is generated once per machine, lives at ~/.supersonic/keys/, and never leaves it; the public key travels inside every receipt, so sonic verify-receipts <path> can check one from any checkout — a teammate's clone, CI, an old backup — with zero setup. What it proves is narrower than a Sigstore-style attestation: not that the prompt itself wasn't misleading, but that the receipt file, byte for byte, was produced by a specific local install and hasn't been edited since. Hand-edit a shipped receipt, or the diff around it, without the private key, and the signature stops verifying. Not a Verify signal — like Review Risk, it never blocks a turn.

What is Risk-Aware Model Escalation?

A feedback loop from Review Risk back into Build and Verify, not a new gate. After a turn ships, if its Review Risk brief contains at least one HIGH-risk file (large blast radius, a sensitive-path match, or no matching test change), the next turn — and only that one — runs the coding-agent CLI and Supersonic's own critic call at a stronger model. Each coding agent gets its own configured escalation target (a verified flag: --model for Claude Code/Codex/OpenCode/Aider, --model for Cursor Agent's CLI), and the critic call uses your provider's own default (strongest) model instead of its normal fast model. It decays automatically the turn after — there's no standing state, just "was the immediately preceding turn's shipped diff flagged high-risk." If you haven't configured an escalation model for the agent you're using, or the previous turn had no high-risk files, nothing changes and the loop runs at its normal cost.

Does Black Box Replay's "recompute in your browser" button actually run independently, or is it just decoration?

It actually runs — it's the Web Crypto API's crypto.subtle.digest('SHA-256', ...), hashing the exact diff text embedded in the page and comparing it to that turn's receipt, live, in your browser, with zero trust placed in whatever generated the page. What's decoration-free about it is also what limits it: the button is only ever shown for a turn where the page-generation step has already verified, server-side, that the diff it reconstructed from git history hashes to the exact value the receipt was signed over. If a turn had a failed-and-rolled-back turn before it, the reconstructed diff picks up that turn's preserved Continuity Graph entries in a way this feature doesn't attempt to separate out byte-for-byte — so the button is simply not offered for that turn, with a plain explanation, rather than risking a false "verified" claim.

Does PTY-native execution actually intercept file writes before they hit disk?

No, and Supersonic doesn't claim it does. A pseudo-terminal (PTY) governs a child process's stdin/stdout stream — that's it. It cannot see, let alone intercept, a filesystem write syscall; that's a fundamentally different, much heavier mechanism (ptrace/seccomp syscall interception, or a FUSE overlay filesystem), and this feature doesn't implement either. What PTY-native execution actually does: run the coding-agent CLI inside a real pty.fork()ed terminal instead of a plain subprocess pipe, so a CLI that behaves differently when it detects it isn't attached to a terminal (many disable colors, progress bars, or interactive confirmations the instant stdout is a pipe) runs the same way it would for a human typing the command directly. It's off by default, POSIX-only, and falls back to the standard subprocess path automatically on any platform or CLI it can't run cleanly on.

Does Docker Sandbox stop a coding agent from reaching the internet?

No — and that's deliberate, not an oversight. The coding agent still needs outbound network access to reach its own LLM provider's API (Anthropic, OpenAI, whichever it's configured for); sandboxing that away would break the agent's basic function, not just contain its blast radius. What Docker Sandbox actually contains is the filesystem: the container only has the project's own workdir bind-mounted in, runs with all Linux capabilities dropped, and is memory/CPU/PID-capped, so a runaway or hallucinating shell command can trash the container's own throwaway filesystem but can't reach anything else on the host. It also isn't zero-config the way PTY mode is — it requires Docker installed and a real image built or pulled ahead of time (a reference Dockerfile covering Claude Code, Codex, and Aider ships in the repo). No Docker install, no configured image, or an unreachable daemon all fall back cleanly to PTY or plain-subprocess execution — a missing optional dependency never fails a build turn.

What is Live Syntax Watch, and how is it different from PTY execution?

A separate, unrelated mechanism — not something derived from the PTY. It's a background thread that polls every touched Python file's modification time roughly every 250ms and re-parses it with ast.parse the instant it changes, so a broken file gets flagged within a fraction of a second of being saved instead of only at the end of the whole turn (when Syntax Shield's diff-based check already catches it). A file's first-ever sighting is never flagged, so a pre-existing broken file the agent hasn't touched yet is never misreported. It's observability, not enforcement: it doesn't pause or kill the agent process, because doing that safely and generally across five different third-party CLIs isn't something this feature attempts.

Is the Self-Evolving Rules Engine the same as OpenEvolve/EvoAgentX-style prompt mutation?

No — deliberately narrower. Those projects run genetic-algorithm-style search over a population of candidate prompts scored against a fitness function. Supersonic's version is simpler: once the exact same Verify failure category (a syntax error, a hallucinated dependency, a failing test, a weak-coverage mutant, etc.) has repeated across a configurable number of turns, one supervisor-critic LLM call writes exactly one concise rule from that specific failure trace and appends it — never rewrites or randomly mutates an existing one — to the project's own .supersonic/rules.md, which then gets folded into every subsequent turn's prompt. Best-effort, it also mirrors into .cursorrules or CLAUDE.md if the project already has one of those real convention files, updating only its own clearly marked section — never creating one of those files from scratch, since that would silently opt a project into a tool convention it never asked for.

What does Multi-Repository State Anchoring actually coordinate?

Checkpoint and rollback, across however many independent git working directories you register for a project — not code generation. The coding agent still runs exactly once per turn, against the primary repo only, exactly as it does today; nothing here spins up a second agent run against a linked frontend or backend repo automatically. What's real: register N repo paths once, and from then on every turn snapshots all of them before the agent runs. Once the primary repo's turn is scored, every linked repo is either checkpointed to match (the primary turn passed) or rolled back to its last coordinated anchor (the primary turn failed) — together, never one without the others. That's the actual property a multi-repo feature ticket needs: nothing in the linked set can silently drift out of sync with what the primary repo's Verify gate actually approved.

Is this hosted?

No. Supersonic runs entirely on your machine. This website is documentation and overview only.

Where do API keys go?

~/.supersonic/config.json — never uploaded anywhere. Configure via the local UI at sonic serve.

Do I need four different vendor accounts to try it?

No. One LLM key (Anthropic, OpenAI, or a local Ollama server) is all that's required. GitHub shipping, Linear, and research enrichment are all optional and off by default.

Can I try without API keys?

Yes — sonic run --demo runs the full loop in demo mode with synthetic turns, so you can see the checkpoint/verify/rollback mechanics before connecting a provider.
