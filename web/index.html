The build loop that only moves forward on evidence.

Local, open source, one API key. Every turn is committed only if it passes a four-signal Verify gate — otherwise it's rolled back automatically and the failure is written into a structured Continuity Graph so the next attempt doesn't repeat it. Optionally race two coding agents against each other and let a bandit learn which one wins at your kind of work.

4/4signals gate every turn

Checkpoint → Verify → Rollback.

A turn is committed only if enough of the four signals that actually ran come back positive. A failed turn is hard-reset to the last verified state, not left to compound.

  • Invariantsalways kept
  • Failuresalways kept
  • Decisionsranked
  • Old entriesdistilled

A Continuity Graph, not a truncated transcript.

Structured decisions, invariants, and failures, retrieved by relevance to the current turn — not a flat context blob trimmed down to fit a budget.

Bandit-gated Agent Racing.

Turn on racing and two agents compete in isolated git worktrees when the outcome is still uncertain — a Thompson-sampling bandit learns which one wins per task type and stops racing once it knows, so cost decays instead of doubling forever.

Autonomous coding loops fail quietly — they drift, they thrash, they forget what they already tried. Supersonic's loop only keeps what it can prove. No hosted keys. No sponsor-locked integration stack. One API key, your machine, your git history as the safety net.

How it works

1 · Plan

Ground the idea, write the plan

Optional research enrichment, then one call produces a build plan and a name — provider-agnostic, no vendor lock-in.

2 · Checkpoint

Git-native snapshotting

Every verified state is committed and tagged. No custom snapshot format — just git doing what it already does well.

3 · Build

Your agent writes code

Claude Code, Codex, OpenCode, Cursor Agent, or Aider — bring your own CLI, or race two of them.

4 · Verify

Four independent signals

Tests, lint/typecheck, an LLM goal-satisfaction critic, and a diff-similarity thrash detector.

5 · Rollback or ship

Only proven progress sticks

Pass → new checkpoint, pushed to GitHub. Fail → hard reset to the last good state, failure logged to the Continuity Graph.

Supported coding agents

Install any CLI on your PATH. Pick one per project, or race two of them.

Anthropic's agent CLI. Authenticates via the local Claude Code login.

Codex

OpenAI Codex CLI — codex exec --full-auto. Authenticates via the local Codex login.

OpenCode

Open-source coding agent. Runs from a generated prompt file in your project dir.

Cursor Agent

Cursor's local agent CLI when installed — same loop, your subscription.

Aider

The original terminal pair-programmer — works as a drop-in racing partner.

Quickstart

Clone, install, onboard, build.

# Install
git clone https://github.com/your-org/supersonic.git
cd supersonic
./bootstrap.sh
source .venv/bin/activate
sonic serve
# → http://127.0.0.1:8787 → Open Supersonic → onboard → Build

# Headless
sonic run --idea "CLI that syncs issues to markdown" --agent claude
sonic run --agent claude --race --race-with codex
sonic run --demo
sonic doctor

FAQ

Is this hosted?

No. Supersonic runs entirely on your machine. This website is documentation and overview only.

Where do API keys go?

~/.supersonic/config.json — never uploaded anywhere. Configure via the local UI at sonic serve.

Do I need four different vendor accounts to try it?

No. One LLM key (Anthropic, OpenAI, or a local Ollama server) is all that's required. GitHub shipping, Linear, research, and racing are all optional and off by default.

Won't Agent Racing double my API costs?

Only while the bandit is still uncertain which agent wins at a given task type. It's Thompson sampling, not always-on racing — the frequency of races decays as it learns, plus a hard per-run ceiling caps the worst case. Off by default either way.

Can I try without API keys?

Yes — sonic run --demo runs the full loop in demo mode with synthetic turns, so you can see the checkpoint/verify/rollback mechanics before connecting a provider.
