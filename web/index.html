How it works

1 · Plan

Ground the idea, write the plan

Optional research enrichment, then one call produces a build plan and a name — provider-agnostic, no vendor lock-in.

2 · Checkpoint

Git-native snapshotting

Every verified state is committed and tagged. No custom snapshot format — just git doing what it already does well.

3 · Build

Your agent writes code

Claude Code, Codex, OpenCode, Cursor Agent, or Aider — bring your own CLI.

4 · Dependency Trust

Every new package checked before it ships

Newly-added dependencies are verified against the real PyPI/npm registry. A package that doesn't exist fails the turn — one corrective re-prompt, then rollback, before it ever reaches the expensive gate below.

5 · Secret Leak

Every diff scanned for hardcoded credentials

Added lines are checked against real credential formats (AWS, GitHub, Slack, Stripe, Anthropic, OpenAI, Google, PEM keys) and new .env files. A high-confidence match fails the turn — one corrective re-prompt, then rollback, same as Dependency Trust.

6 · Verify

Seven independent signals

Tests, lint/typecheck, an LLM goal-satisfaction critic, a diff-similarity thrash detector, an optional browser telemetry check, and the Dependency Trust and Secret Leak results above.

7 · Rollback or ship

Only proven progress sticks

Pass → new checkpoint, pushed to GitHub. Fail → hard reset to the last good state, failure logged to the Continuity Graph.

8 · Review Risk

Told what to actually check

Every shipped turn's changed files ranked by blast radius, sensitive-path exposure, missing test coverage, unverified dependencies, and suspicious credential-shaped assignments — a static heuristic, not another LLM call, so it's instant.

FAQ

What is Review Risk, actually?

A static heuristic (no LLM call) that runs once a turn ships. It parses the diff, counts how many other files import each changed file (blast radius), checks changed lines and paths against a list of sensitive-surface keywords (auth, payment, permissions, migrations, deletion), and checks whether a matching test file was touched in the same turn. Files score high/medium/low and get a plain-English reason. It has false positives and false negatives — the goal is redirecting a human's five minutes of attention, not replacing judgment.

What is the Dependency Trust Gate?

A pre-flight check that runs before the expensive Verify gate. It reads the turn's diff for any newly-added package (manifest entries in requirements.txt/pyproject.toml/Pipfile/package.json, or a raw install-command line), and checks each one against the real PyPI or npm registry. A package that doesn't exist fails the turn outright — that's "slopsquatting": an attacker pre-registers a name an AI agent is statistically likely to hallucinate, ships malware under it, and waits. It can't stop a malicious install from executing once inside a single turn's working tree (Supersonic doesn't sandbox the coding agent's own shell commands), but it guarantees that dependency never gets checkpointed, shipped, or pushed to GitHub — and the agent gets one corrective re-prompt naming the exact bad package first.

What is the Secret Leak Gate?

A pre-flight check, same family as Syntax Shield and the Dependency Trust Gate, that runs before the expensive Verify gate. It scans only the added lines of a turn's diff for the structural shape of a real credential: AWS access key IDs, PEM private key blocks, GitHub/Slack/Stripe/Anthropic/OpenAI/Google token formats, and Slack webhook URLs. A match on any of those formats is specific enough to be a real secret, not a coincidence, and fails the turn outright — one corrective re-prompt naming the exact file and pattern, then rollback if it's still there. A new or modified real .env file (not .env.example) is flagged the same way regardless of content. A separate, noisier heuristic — a variable name like api_key or password assigned a long opaque string — is surfaced in the Review Brief instead of blocking, since it's prone to false positives on things like test fixture IDs. GitGuardian's State of Secrets Sprawl 2026 report found AI-assisted commits leak a real secret at roughly double the rate of human-only commits (~3.2% vs. ~1.6%), and that 64% of secrets confirmed valid in 2022 were still exploitable in January 2026 because nobody rotated them — this gate exists to stop that pattern before it starts.

Is this hosted?

No. Supersonic runs entirely on your machine. This website is documentation and overview only.

Where do API keys go?

~/.supersonic/config.json — never uploaded anywhere. Configure via the local UI at sonic serve.

Do I need four different vendor accounts to try it?

No. One LLM key (Anthropic, OpenAI, or a local Ollama server) is all that's required. GitHub shipping, Linear, and research enrichment are all optional and off by default.

Can I try without API keys?

Yes — sonic run --demo runs the full loop in demo mode with synthetic turns, so you can see the checkpoint/verify/rollback mechanics before connecting a provider.
